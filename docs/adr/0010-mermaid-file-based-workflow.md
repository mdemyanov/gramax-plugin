# ADR-0010: Mermaid skill — file-based workflow

**Status:** Accepted
**Date:** 2026-05-12
**Plugin:** gramax

## Context

Spec `docs/superpowers/specs/2026-05-12-mermaid-file-based-design.md` описывает фичу **mermaid-file-based v4.0.0**:

Текущий `gramax:mermaid` (v3.0.0) вставляет DSL **inline** в md-файл — либо как `<mermaid>…</mermaid>` (XML-syntax), либо как ` ```mermaid … ``` ` (Markdown-syntax). Это противоречит реальному формату Gramax, задокументированному в `plugins/gramax/skills/writer/references/blocks.md` (строки 178–180):

```
<mermaid path="./diagram.mermaid" width="800px" height="450px"/>
```

Gramax-рендер ожидает самозакрывающийся тег-ссылку на внешний `.mermaid`-файл. Inline-блоки, которые генерирует skill сейчас, не отображаются корректно без ручного исправления.

Spec BA закрывает FR-001—FR-008, NFR-001—NFR-005 и формулирует 5 открытых вопросов (Q-001—Q-005), требующих архитектурного решения SA.

**Связь с предшествующими ADR:**

- **ADR-0006** (Active) — semver-policy для marketplace; применяется в Решении 1.
- **ADR-0008** (Accepted) — устанавливает minor/major-семантику в контексте удаления skill'ов; прецедент применяется здесь.
- **ADR-0009** (Accepted) — описывает текущее состояние `gramax:mermaid` (v3.0.0): inline-генерация сохранена без изменений; description уточнён. Данный ADR-0010 частично supersedes ADR-0009 в части описания поведения mermaid (inline → file-based), не затрагивая решения по drawio-stub и submodule.

## Решение 1 — Q-001: Версия bump: major 4.0.0

**Решение:** поднять версию плагина до **4.0.0**; `marketplace.json metadata.version` — до **4.0.0** (синхронно, конвенция ADR-0006).

**Обоснование по ADR-0006 (semver-policy, Active):**

ADR-0006 зафиксировал семантику:
- **Minor (x.N.0)** — additive change: новый skill, новая команда; существующий workflow не нарушается.
- **Major (N.0.0)** — breaking change: удаление skill, несовместимое изменение поведения, смена публичного контракта.

Изменение inline-workflow на file-based является breaking change по двум критериям:

1. **Поведение skill'а изменяется несовместимо.** Пользователи, вызывавшие `gramax:mermaid`, ожидали получить DSL, вставленный непосредственно в md-файл (один артефакт). Теперь skill создаёт `.mermaid`-файл (Write tool) и вставляет тег-ссылку. Workflow пользователя усложняется: появляется новый файл, который нужно учитывать в git-commit и который ссылочно связан с md-страницей.
2. **Контракт вывода skill'а меняется.** ADR-0008 применял этот принцип к удалению skills; здесь применим тот же критерий: изменение _способа хранения результата_ — не additive, а breaking.

**Контраргумент и ответ на него:** «inline-workflow в v3.0.0 сам по себе был некорректен относительно blocks.md — это исправление дефекта, не breaking change». Ответ: с точки зрения пользователя v3.0.0 дефект не был задокументирован как дефект (CHANGELOG 3.0.0 описывает inline-поведение как рабочее). Пользователи могли строить workflow на inline-вставке (например, с учётом, что Gramax их системы поддерживает inline через дополнительные настройки). Major bump корректно сигнализирует о смене контракта.

**Синхронный bump обоих файлов в одном коммите (конвенция ADR-0006):**

| Файл | Было | Станет |
|------|------|--------|
| `plugins/gramax/.claude-plugin/plugin.json` → `version` | `"3.0.0"` | `"4.0.0"` |
| `.claude-plugin/marketplace.json` → `metadata.version` | `"3.0.0"` | `"4.0.0"` |

## Решение 2 — Q-002: Naming convention для .mermaid-файлов

**Решение:** принять BA-предложение `<page-slug>-<diagram-slug>.mermaid` с уточнениями edge cases ниже. Это — закон для Dev, не эвристика.

**Полная спецификация (Dev реализует строго по этим правилам):**

### Правило формирования `<page-slug>`

- Берётся имя md-файла (`target_page`) без расширения.
- Специальный кейс: если файл называется `_index.md` — берётся имя **родительской директории**.
- Примеры:
  - `docs/auth/overview.md` → `page-slug = overview`
  - `docs/auth/_index.md` → `page-slug = auth`
  - `docs/getting-started.md` → `page-slug = getting-started`

### Правило формирования `<diagram-slug>`

- Skill извлекает тематическое имя из запроса пользователя: ключевые существительные процесса/системы.
- Kebab-case, только ASCII, не более 30 символов (ограничение spec BA).
- Если slug длиннее 30 символов — обрезать до 30, сохраняя целые слова (не обрезать посреди слова).
- Если пользователь не назвал тему диаграммы явно — использовать `diagram` как fallback: итоговое имя файла `<page-slug>-diagram.mermaid`.
- Skill **предлагает** имя файла и **спрашивает подтверждение** (FR-002) — пользователь может указать альтернативу.

### Транслитерация кириллицы (Q-002, открытый вопрос spec)

**Решение:** skill **не транслитерирует автоматически**. Если diagram-slug содержит кириллицу (пользователь назвал тему по-русски), skill предлагает latin-вариант на основе смыслового перевода, а не механической транслитерации.

**Обоснование:** `slugify.py` существует в плагине (`plugins/gramax/scripts/slugify.py`) и выполняет транслитерацию, однако NFR-003 запрещает внешние зависимости в skill'е (только Write/Edit/Read tools). Skill является markdown-промптом без скриптовых вызовов. Автоматическая транслитерация в prompt-based skill ненадёжна. Семантический перевод через LLM надёжнее: «последовательность входа» → `login-sequence`, не `posledovatelnost-vkhoda`.

**Алгоритм:** если в запросе — кириллица, skill предлагает два варианта имени файла: один — семантический перевод на английский, второй — запрос у пользователя ввести slug самостоятельно. Пользователь выбирает.

### Запрещённые символы

Символы `:`, `*`, `?`, `"`, `<`, `>`, `|`, `\` — заменяются на `-` (правило из spec).

Пробелы в slug — заменяются на `-`.

### Обработка коллизии имён

Если файл с предложенным именем уже существует (FR-004):

1. Skill читает первые 5 строк существующего файла (Read tool).
2. Выводит содержимое пользователю.
3. Предлагает три опции: перезаписать / выбрать другое имя / отменить.
4. Без явного «перезаписать» — не трогает файл (NFR-001: идемпотентность).

### Примеры (обязательны для Dev)

| md-файл | Тема запроса | Предложенное имя |
|---------|--------------|-----------------|
| `docs/auth/overview.md` | «процесс авторизации» | `overview-auth-flow.mermaid` |
| `docs/auth/_index.md` | «архитектура модуля» | `auth-module-architecture.mermaid` |
| `docs/api/endpoints.md` | (не указана) | `endpoints-diagram.mermaid` |
| `docs/deploy/k8s-setup.md` | «deployment pipeline» | `k8s-setup-deployment-pipeline.mermaid` |
| `docs/main.md` | «очень длинное название процесса обработки данных» | `main-data-processing-pipeline.mermaid` (≤30 символов slug) |

## Решение 3 — Q-003: Backward-compat scope: ПРЕДЛАГАЕТ миграцию, не выполняет автоматически

**Решение:** при обнаружении inline-блока (`<mermaid>…</mermaid>` или ` ```mermaid … ``` `) в `target_page` skill **выводит предупреждение и предлагает миграцию**, но **не выполняет её без явного подтверждения**.

**Обоснование через принцип минимального сюрприза:**

Пользователь вызвал skill для создания **новой** диаграммы, не для миграции существующих. Автоматическая миграция существующего контента в том же вызове нарушает принцип единственной ответственности вызова: пользователь не просил трогать уже существующие блоки. Неожиданное изменение структуры файла — «наименее удивительное» поведение здесь: предупредить, но не действовать.

Дополнительно: автоматическая миграция может быть деструктивной, если inline-блок намеренно оставлен в таком формате (например, для нестандартного рендера в локальной копии Gramax). Запрос подтверждения оставляет решение за пользователем.

**Формат предупреждения (точный текст для SKILL.md):**

```
Обнаружен inline-блок <mermaid>…</mermaid> в файле — это устаревший формат.
Gramax ожидает тег-ссылку <mermaid path="…"/>.
Предлагаю миграцию: извлечь DSL в файл <page-slug>-existing.mermaid и заменить блок на тег.
Выполнить миграцию? (да / нет / пропустить и создать новую диаграмму)
```

**Отдельный `gramax:mermaid-migrate` skill (Q-003 spec):** не создаётся в рамках данной фичи. Обоснование: пакетная миграция — отдельный JTBD, требует отдельного spec и ADR. Добавить как фичу Phase 2 при наличии реального запроса от пользователей. Per-file миграция по подтверждению (FR-006) — достаточный scope для v4.0.0.

## Решение 4 — Q-004: Валидация DSL — только checklist в SKILL.md, без отдельного скрипта

**Решение:** формальная валидация DSL перед записью файла реализуется исключительно через checklist в SKILL.md (FR-003). Отдельный скрипт или grep-валидатор не создаётся.

**Обоснование:**

1. **NFR-003 (без внешних зависимостей):** skill работает только через Write/Edit/Read tools. Вызов bash-скрипта противоречит NFR-003.
2. **NFR-004 (токен-бюджет):** добавление внешнего скрипта (даже если запускать через Bash tool) требует промпт-инструкций по вызову, обработке stdout и интерпретации ошибок — токен-нагрузка растёт.
3. **Checklist покрывает те же паттерны.** Пять критических правил синтаксиса в SKILL.md (list-syntax, subgraph naming, node references, special chars, arrows) уже описаны как checklist. LLM-самопроверка по чеклисту является достаточным барьером для типовых ошибок парсера.
4. **Единственный реальный выигрыш скрипта:** детерминированная проверка паттерна `[0-9]\. ` в DSL (AC-010). Этот паттерн **уже входит** в checklist SKILL.md («Нет паттерна `число. пробел` в тексте нод»). Дублировать в скрипт — избыточно.

**Что добавляется в SKILL.md (расширение чеклиста для file-based workflow):**

Добавить пункт к существующему checklist'у:
- `[ ]` Содержимое `.mermaid`-файла не содержит Gramax-разметки: нет `<mermaid>`, нет ` ```mermaid ` — только чистый DSL.

Это закрывает AC-007 (spec).

## Решение 5 — Q-005: Height по умолчанию — фиксированный 450px

**Решение:** для MVP использовать фиксированный дефолт `height="450px"`, адаптивная эвристика не реализуется.

**Обоснование:**

1. **Ground-truth из blocks.md:** в `blocks.md` (строки 178–180) указано `width="800px" height="450px"` — это значения, которые Gramax-документация устанавливает как эталон. Отклоняться от документированного стандарта без явной причины — нежелательно.
2. **Эвристика по числу нод ненадёжна.** Количество нод не определяет высоту однозначно: глубина дерева, количество уровней вложенности, наличие swimlanes, горизонтальное vs вертикальное направление — всё это влияет на реальный размер. Эвристика будет давать ложные срабатывания и потребует сложного prompt-кода.
3. **Пользователь может переопределить явно.** FR-005 и UX-параметры spec предусматривают явное указание `width`/`height` в запросе. Дефолт — это fallback, не принудительное ограничение.

**Конвенция для Dev (точные значения по умолчанию):**

```
width="800px" height="450px"
```

Если пользователь указал только один параметр — второй берётся из дефолта.

## Consequences

**Положительные:**
- Skill приводится в соответствие с ground-truth форматом Gramax (`blocks.md`): диаграммы отображаются корректно без ручного исправления.
- DSL хранится в отдельном `.mermaid`-файле — версионируется и редактируется независимо от md-страницы.
- Naming convention создаёт предсказуемую структуру файлов в каталоге.
- Принцип минимального сюрприза соблюдён: миграция существующих блоков — только по подтверждению.

**Отрицательные:**
- Breaking change для пользователей v3.0.0: workflow усложняется (появляется файл `.mermaid`, который нужно включать в git-commit).
- Skill создаёт два артефакта вместо одного — пользователи, привыкшие к inline, должны адаптировать процессы ревью и публикации.
- Кириллические темы диаграмм требуют дополнительного диалога (без автотранслитерации).

**Mitigations:**
- CHANGELOG.md § 4.0.0 содержит Migration guide с описанием нового workflow и edge cases.
- Major bump (4.0.0) явно сигнализирует о breaking change.
- Предложение имени файла + запрос подтверждения (FR-002) снижает friction при работе с naming convention.
- Граф edge cases (коллизия имён, кириллица, `_index.md`) покрыт явными инструкциями в SKILL.md.

## Alternatives Considered

**Q-001: Minor bump 3.1.0 вместо major 4.0.0**
Отклонено. ADR-0006 явно ограничивает minor случаями additive change. Смена способа хранения результата (inline → file-based) является несовместимым изменением контракта поведения skill'а. Minor bump создаст у пользователей ложное ощущение совместимости.

**Q-002: Автоматическая транслитерация кириллицы через slugify.py**
Отклонено. NFR-003 запрещает внешние зависимости и скриптовые вызовы в skill'е. Вызов `slugify.py` нарушает архитектурный принцип. Семантический перевод через LLM семантически точнее механической транслитерации.

**Q-002: Хранение .mermaid-файлов в поддиректории assets/**
Не рассматривалось в spec, отклонено как архитектурное решение. Хранение рядом со страницей — инвариант из spec BA (раздел «Бизнес-правила»): «файл `.mermaid` хранится рядом со страницей — не в отдельной папке assets». Инвариант принят без изменений.

**Q-003: Автоматическая миграция inline-блоков без подтверждения**
Отклонено. Нарушает принцип минимального сюрприза. Пользователь вызвал skill для создания новой диаграммы, а не для рефакторинга существующих. Автоматическая мутация существующего контента без явного запроса — деструктивна и непредсказуема.

**Q-003: Создание отдельного skill `gramax:mermaid-migrate` для пакетной миграции**
Отложено как Phase 2. Пакетная миграция — отдельный JTBD, требует отдельного spec. Добавлять в scope v4.0.0 нецелесообразно.

**Q-004: Grep-скрипт для валидации запрещённых паттернов DSL**
Отклонено. NFR-003 запрещает внешние зависимости. Паттерны, которые покрыл бы скрипт, уже перечислены в checklist SKILL.md. Дублирование в скрипт повышает сложность без значимого прироста надёжности.

**Q-005: Адаптивный height по числу нод/строк DSL**
Отклонено для MVP. Эвристика ненадёжна (размер диаграммы зависит не только от числа нод). Пользователь может указать `height` явно в запросе. Фиксированный дефолт из ground-truth (`blocks.md`) является наиболее безопасным выбором.

## Контракт с QA-author

**AC (полный список из spec):**
- AC-001: `.mermaid`-файл создаётся в директории `target_page`
- AC-002: первая строка `.mermaid`-файла — один из 8 поддерживаемых типов
- AC-003: `target_page` содержит тег `<mermaid path="./"…/>`
- AC-004: тег содержит атрибуты `width` и `height`
- AC-005: тег содержит дефолты `width="800px" height="450px"` при отсутствии явного указания
- AC-006: тег самозакрывающийся (без содержимого между тегами)
- AC-007: `.mermaid`-файл не содержит `<mermaid>` и fenced block ` ```mermaid `
- AC-008: повторный вызов с тем же именем без подтверждения → файл не изменён
- AC-009: при inline-блоке в `target_page` stdout содержит «устаревший» или «migration»
- AC-010: DSL в `.mermaid`-файле не содержит паттерн `[0-9]\. ` (list-syntax conflict)
- AC-011: SKILL.md не содержит «inline DSL, без файла»

**Архитектурный контекст:**
- Скоуп: `plugins/gramax/skills/mermaid/SKILL.md`, `plugins/gramax/skills/mermaid/references/syntax-rules.md`
- External boundaries: файловая система (Write tool создаёт `.mermaid`-файл; Edit tool вставляет тег в md), Read tool (чтение `.doc-root.yaml`, чтение существующего `.mermaid`-файла при коллизии)
- Нет MCP, нет bash-скриптов, нет сетевых вызовов

**Edge cases / boundary conditions:**
- `target_page = _index.md` → `page-slug` берётся из родительской директории (проверять отдельно от стандартного кейса)
- Файл `.mermaid` с предложенным именем уже существует → skill не перезаписывает без подтверждения (AC-008); содержимое файла после вызова с ответом «нет» — без изменений
- Inline-блок `<mermaid>…</mermaid>` в target_page → AC-009 (smoke, подстановка преднамеренно некорректного файла)
- Inline fenced block ` ```mermaid … ``` ` в target_page → аналогично AC-009 (отдельный smoke-тест)
- `diagram-slug` длиннее 30 символов → обрезка до 30, не посреди слова
- Явный `height=600px` в запросе → тег содержит `height="600px"`, не дефолт (AC-005 не срабатывает)
- `width` указан, `height` не указан → `height="450px"` дефолт (AC-005 частично)
- Кириллица в теме диаграммы → skill предлагает latin-вариант, не падает с ошибкой
