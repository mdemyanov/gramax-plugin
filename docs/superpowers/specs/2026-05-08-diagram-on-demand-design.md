# Diagram on Demand

## JTBD

Когда разработчик документирует архитектуру в Gramax-каталоге, он хочет явно попросить «нарисуй mermaid-диаграмму X» или «сделай drawio-схему Y» и получить готовый файл, вставленный в нужную md-страницу в соответствии с Gramax-соглашениями о хранении вложений, чтобы не тратить время на ручное создание DSL, конвертацию и встраивание ссылки.

---

## Описание

Сейчас `gramax:diagrams` skill описывает **правила** работы с диаграммами, но не позволяет пользователю явно запустить генерацию диаграммы по описанию. Пользователь должен сам сформулировать DSL (mermaid-syntax или drawio-XML), запустить конвертацию вручную и вставить ссылку в md. Это ручная, ошибко-ёмкая цепочка.

Новая фича `diagram-on-demand` добавляет два skill'а (или один агент — финальное архитектурное решение за SA): пользователь говорит «нарисуй mermaid flowchart авторизации» или «создай drawio-схему CI/CD», модель генерирует DSL/XML, вызывает нужный бэкенд (MCP mermaid или MCP drawio), сохраняет результат рядом с целевой md-страницей и добавляет ссылку по синтаксису каталога (XML или Markdown — из `.doc-root.yaml`).

**Движок выбирает пользователь явно** — никакого угадывания по контексту. Это исключает неопределённость при выборе mermaid vs drawio и позволяет тестировать каждый путь независимо.

**Зависимость от MCP-бэкендов:** mermaid-путь использует vendored `claude-mermaid` MCP (`mermaid_preview` / `mermaid_save`). Drawio-путь использует один из трёх кандидатов MCP-серверов — выбор конкретного варианта (`jgraph/drawio-mcp`, `drawio-mcp` PyPI, `lgazo/drawio-mcp-server`) является предметом ADR от SA. `drawio_convert.py` (уже существует) встраивает XML в SVG для round-trip редактирования.

## User Journeys

### Journey 1 — Mermaid flowchart с нуля

Пользователь открывает Claude Code в рабочей директории Gramax-каталога и говорит:

> «Нарисуй mermaid flowchart процесса авторизации: пользователь вводит логин/пароль, система проверяет токен, при успехе — redirect на dashboard, иначе — сообщение об ошибке. Сохрани рядом с `docs/auth/login-flow.md`.»

Ожидаемое поведение:

1. Skill читает `.doc-root.yaml` из корня каталога (`docs/`) → определяет `syntax: XML` или `syntax: Markdown`.
2. Генерирует mermaid DSL (тип `flowchart TD`, не из запрещённого списка).
3. Вызывает MCP mermaid (`mermaid_save`) с путём `docs/auth/login-flow.svg` (или `.png` — зависит от того, что даёт MCP; уточнит SA).
4. Вставляет в `docs/auth/login-flow.md` ссылку по синтаксису каталога.
5. Выводит в stdout итог: путь к файлу и вставленный фрагмент.

Команда-триггер (пример): «нарисуй mermaid [описание]» / «создай mermaid-диаграмму [описание]» / «добавь mermaid на страницу [path]».

### Journey 2 — Drawio-схема с нуля

Пользователь говорит:

> «Сделай drawio-схему деплоя: три сервиса (API, Worker, DB), стрелки зависимостей. Сохрани рядом с `docs/infra/deploy.md`.»

Ожидаемое поведение:

1. Skill определяет синтаксис каталога через `.doc-root.yaml`.
2. Генерирует drawio XML (mxfile-формат).
3. Вызывает MCP drawio-бэкенд для получения SVG-рендера (или выполняет `drawio_convert.py` локально — зависит от выбранного SA MCP-варианта).
4. Сохраняет `docs/infra/deploy.drawio` и `docs/infra/deploy.svg` (с embedded XML через `drawio_convert.py`).
5. Вставляет `<Image src="deploy.svg" />` (XML syntax) или `![](deploy.svg)` (Markdown syntax) в `docs/infra/deploy.md`.
6. Выводит итог в stdout.

Команда-триггер: «сделай drawio [описание]» / «нарисуй drawio-схему [описание]» / «создай drawio на страницу [path]».

### Journey 3 — Повторный вызов при ошибке движка

Пользователь запрашивает mermaid-диаграмму типа `gitGraph`:

> «Нарисуй mermaid gitGraph нашего release-процесса.»

Ожидаемое поведение:

1. Skill проверяет тип диаграммы против списка поддерживаемых Gramax-типов.
2. Обнаруживает, что `gitGraph` не поддерживается.
3. Выводит в stdout предупреждение с конкретной причиной и предлагает альтернативу (`flowchart` или drawio).
4. Не создаёт файлы, не вставляет ссылки.

## Функциональные требования

- **FR-001:** при триггере «нарисуй mermaid [описание]» skill генерирует mermaid DSL одного из поддерживаемых типов (`flowchart`, `sequenceDiagram`, `gantt`, `classDiagram`, `stateDiagram-v2`, `erDiagram`, `pie`, `mindmap`) и сохраняет результат рядом с указанной md-страницей.
- **FR-002:** при триггере «сделай drawio [описание]» skill генерирует drawio XML и сохраняет `.drawio` + `.svg` (с embedded XML) рядом с указанной md-страницей.
- **FR-003:** skill читает `.doc-root.yaml` из корня указанного каталога и использует значение `syntax` для выбора формата ссылки (`<Image src="..."/>` для XML, `![](...)` для Markdown).
- **FR-004:** skill вставляет ссылку на диаграмму в целевой md-файл в позицию, указанную пользователем (или в конец файла по умолчанию).
- **FR-005:** при запросе неподдерживаемого mermaid-типа (`gitGraph`, `journey`, `requirementDiagram`, `C4Context`) skill выводит предупреждение с именем типа и причиной, не создаёт файлы.
- **FR-006:** если `.doc-root.yaml` отсутствует в указанном каталоге, skill ищет его в родительских директориях (до корня проекта); если не найден — использует `Markdown`-синтаксис как дефолт и выводит предупреждение.
- **FR-007:** при наличии уже существующего файла с тем же именем skill предупреждает пользователя и не перезаписывает без подтверждения.
- **FR-008:** имена создаваемых файлов формируются как slug (ASCII-only через `slugify.py`) — кириллица в именах файлов не допускается.
- **FR-009:** drawio-путь использует `drawio_convert.py` для генерации SVG с embedded XML (round-trip редактирование должно быть сохранено).
- **FR-010:** mermaid-путь вызывает MCP `mermaid_save` из `claude-mermaid` с явным путём сохранения внутри Gramax-каталога.
- **FR-011:** если MCP drawio-бэкенд недоступен, skill сообщает об этом в stderr и завершается с exit code 1; `.drawio`-файл всё равно сохраняется (для ручной конвертации).
- **FR-012:** если MCP mermaid-бэкенд недоступен, skill выводит сгенерированный DSL в stdout с инструкцией по ручному сохранению.

## Нефункциональные требования

- **NFR-001:** полный цикл «генерация DSL → MCP-вызов → сохранение файла → вставка ссылки» должен завершаться не более чем за 30 секунд при доступном MCP-бэкенде (вместе с LLM-генерацией DSL).
- **NFR-002:** токен-бюджет системного промпта skill'а — не более 2000 токенов (исключая runtime-контекст пользователя).
- **NFR-003:** skill работает на macOS и Linux (bash 3.2+, Python 3.10+).
- **NFR-004:** скрипты не зависят от сторонних Python-пакетов за пределами stdlib (кроме зависимостей MCP-серверов, управляемых SA).
- **NFR-005:** при ошибке MCP skill не оставляет незавершённых файлов-«заглушек» в каталоге (атомарность операции записи).

## UX / Интерфейс

**Триггеры skill'а (natural language):**
- «нарисуй mermaid [тип] [описание]»
- «создай mermaid-диаграмму [описание] для [путь к странице]»
- «сделай drawio-схему [описание] рядом с [путь к странице]»
- «добавь drawio [описание] на [путь к странице]»

**Параметры, которые skill извлекает из запроса:**
- `engine` — явный выбор пользователя: `mermaid` или `drawio` (обязательный; при отсутствии — переспросить).
- `description` — словесное описание диаграммы (обязательный).
- `target_page` — путь к md-файлу, рядом с которым сохраняется диаграмма (обязательный; при отсутствии — переспросить).
- `diagram_name` — имя файла без расширения (опциональный; по умолчанию — slug из первых слов description).
- `diagram_type` — тип mermaid (`flowchart`, `sequenceDiagram`, etc.) (опциональный для mermaid; skill выбирает по смыслу description).

**Формат вывода:**
- Успех: в stdout — созданные пути файлов + вставленный фрагмент md.
- Предупреждение: в stdout с префиксом `[WARN]`.
- Ошибка: в stderr с префиксом `[ERROR]` + exit code 1.

**Пример успешного вывода:**
```
Created: docs/auth/login-flow.svg
Inserted into docs/auth/login-flow.md:
  <Image src="login-flow.svg" alt="Поток авторизации" />
```

## I/O Контракты

### Mermaid-путь

| Элемент | Описание |
|---------|----------|
| Входной триггер | Натуральный язык: «нарисуй mermaid [описание]» |
| Целевая страница | Абсолютный или относительный путь к `.md` файлу |
| Генерируемый DSL | Mermaid-syntax, один из 8 поддерживаемых типов |
| MCP-вызов | `mermaid_save(path=<target_dir>/<name>.svg, content=<dsl>)` |
| Выходной файл | `<target_dir>/<name>.svg` |
| Ссылка в md | `<mermaid>...</mermaid>` (XML syntax) или ` ```mermaid...``` ` (Markdown syntax) |

### Drawio-путь

| Элемент | Описание |
|---------|----------|
| Входной триггер | Натуральный язык: «сделай drawio [описание]» |
| Целевая страница | Абсолютный или относительный путь к `.md` файлу |
| Генерируемый XML | mxfile-формат, валидный drawio XML |
| Сохранение исходника | `<target_dir>/<name>.drawio` |
| Конвертация в SVG | `python3 drawio_convert.py <name>.drawio <name>.svg` |
| MCP-вызов | зависит от выбранного SA drawio-бэкенда |
| Выходные файлы | `<target_dir>/<name>.drawio` + `<target_dir>/<name>.svg` |
| Ссылка в md | `<Image src="<name>.svg" />` (XML) или `![...](<name>.svg)` (Markdown) |

### Edge Cases

| Случай | Ожидаемое поведение |
|--------|-------------------|
| Неподдерживаемый mermaid-тип (`gitGraph`, `journey`, `requirementDiagram`, `C4Context`) | stdout: `[WARN] Тип gitGraph не поддерживается Gramax. Рекомендуется: flowchart или drawio.` Файлы не создаются. Exit code 0. |
| Кириллица в именах файлов | `slugify.py` преобразует в ASCII-slug. Выводится в stdout: `Имя файла: потok-vhoda.svg → login-flow.svg` |
| `.doc-root.yaml` отсутствует в дереве | `[WARN] .doc-root.yaml не найден. Используется синтаксис Markdown по умолчанию.` Продолжает выполнение. |
| MCP drawio-бэкенд не установлен | `[ERROR] MCP drawio-бэкенд недоступен. <name>.drawio сохранён. Для конвертации выполни: python3 drawio_convert.py ...` Exit code 1. |
| Файл уже существует | `[WARN] Файл <path> уже существует. Перезаписать? (y/n)` — ожидает подтверждения. |
| `target_page` не найден | `[ERROR] Файл <path> не существует. Укажи корректный путь к md-странице.` Exit code 1. |
| Пустое описание диаграммы | Skill переспрашивает: «Опиши содержание диаграммы.» |

## Acceptance Criteria

- [ ] **AC-001:** при запросе «нарисуй mermaid flowchart [описание]» с указанием существующей md-страницы в каталоге с `syntax: XML` — создаётся файл `<name>.svg` рядом со страницей и в md-файл вставляется `<mermaid>...</mermaid>` блок.
- [ ] **AC-002:** при запросе «нарисуй mermaid flowchart [описание]» с указанием существующей md-страницы в каталоге с `syntax: Markdown` — в md-файл вставляется fenced ` ```mermaid...``` ` блок.
- [ ] **AC-003:** при запросе «нарисуй mermaid gitGraph [описание]» — в stdout появляется `[WARN]` с именем неподдерживаемого типа, файлы не создаются, exit code = 0.
- [ ] **AC-004:** при запросе «сделай drawio-схему [описание]» с указанием существующей md-страницы — создаются два файла: `<name>.drawio` и `<name>.svg`; `<name>.svg` содержит атрибут `content=` (embedded XML): `grep -c 'content=' <name>.svg` возвращает 1.
- [ ] **AC-005:** при запросе «сделай drawio-схему [описание]» с указанием существующей md-страницы в каталоге с `syntax: XML` — в md-файл вставляется `<Image src="<name>.svg" />`.
- [ ] **AC-006:** если `.doc-root.yaml` отсутствует в дереве каталога — в stdout появляется `[WARN]` про отсутствие файла и используется Markdown-синтаксис по умолчанию; выполнение не прерывается.
- [ ] **AC-007:** при запросе с описанием, содержащим кириллические слова в качестве имени файла — итоговое имя файла содержит только ASCII (через `slugify.py`); в stdout отображается преобразованное имя.
- [ ] **AC-008:** если MCP mermaid-бэкенд (`claude-mermaid`) недоступен — в stdout выводится сгенерированный DSL и инструкция по ручному сохранению; файл `.svg` не создаётся; exit code = 0.
- [ ] **AC-009:** если MCP drawio-бэкенд недоступен — файл `.drawio` сохраняется, файл `.svg` не создаётся; в stderr — `[ERROR]` с командой ручной конвертации; exit code = 1.
- [ ] **AC-010:** если файл с указанным именем уже существует в целевой директории — в stdout появляется `[WARN]` с путём к файлу; перезапись не выполняется без явного подтверждения пользователя.
- [ ] **AC-011:** при запросе без указания целевой md-страницы (`target_page`) — skill переспрашивает пользователя и не создаёт файлы до получения ответа.
- [ ] **AC-012:** созданный `.drawio` файл является валидным XML: `python3 -c "import xml.etree.ElementTree as ET; ET.parse('<name>.drawio')"` завершается с exit code 0.

## Out of Scope (MVP)

- **Auto-detection движка:** движок пользователь указывает явно. Никакого вывода «здесь лучше подойдёт drawio» без запроса.
- **Генерация drawio из XML без MCP-рендера:** если SA решит, что выбранный MCP не поддерживает генерацию XML из описания — генерация drawio с нуля из натурального языка исключается из MVP. В этом случае drawio-путь поддерживает только конвертацию существующего `.drawio` в `.svg`.
- **Редактирование существующей SVG через skill:** пользователь редактирует `.drawio` вручную в diagrams.net, skill не предоставляет команды `edit`.
- **Live-preview в браузере:** preview — функциональность `claude-mermaid` MCP; skill не дублирует это поведение.
- **Batch-генерация нескольких диаграмм:** один вызов = одна диаграмма.
- **Импорт из других форматов** (PlantUML, Lucidchart, Visio): вне скоупа.
- **Генерация диаграмм без привязки к Gramax-каталогу** (произвольный путь вне `.doc-root.yaml` дерева): вне скоупа.
- **GitGraph, journey, requirementDiagram, C4Context в mermaid:** неподдерживаемые типы; не реализуем обходной путь в MVP.

## Open Questions для SA

1. **Split плагина:** один skill `diagram-on-demand` внутри `gramax` или два отдельных плагина `diagram-mermaid` / `diagram-drawio`? Критерий — сложность `plugin.json` и независимость MCP-конфигураций.
2. **Drawio MCP-выбор:** какой из трёх бэкендов (`jgraph/drawio-mcp` Apache-2.0 hosted/stdio, `drawio-mcp` PyPI MIT Python 3.11+, `lgazo/drawio-mcp-server` MIT TS) выбирается и почему? Влияет на FR-011 и AC-009.
3. **MCP mermaid_save сигнатура:** принимает ли `claude-mermaid` MCP путь к файлу напрямую, или только preview + ручное сохранение? Если только preview — mermaid-путь требует дополнительного шага записи файла из Claude Code.
4. **Формат drawio-вывода MCP:** генерирует ли drawio-MCP SVG из описания, или только конвертирует готовый XML? Если только конвертирует — LLM должна генерировать валидный mxfile XML, что увеличивает риск невалидного XML (см. AC-012).
5. **Атомарность записи при ошибке:** нужна ли транзакция (temp-файл → rename) для `drawio_convert.py`, или достаточно удаления недосозданного файла при ошибке?
6. **Позиция вставки ссылки в md:** всегда в конец файла или skill должен угадывать место вставки по контексту страницы? (UX-решение с импликацией на сложность промпта.)

## Бриф для SA

**Spec:** `docs/superpowers/specs/2026-05-08-diagram-on-demand-design.md`

**Спроектировать:**
- Структуру плагина(ов): один `diagram-on-demand` skill в `gramax` или два независимых плагина (`diagram-mermaid`, `diagram-drawio`).
- Конфигурацию `plugin.json` с `mcpServers` для каждого движка.
- Выбор drawio MCP-бэкенда (один из трёх кандидатов) с обоснованием в ADR.
- Сигнатуру MCP-вызовов для mermaid и drawio путей.
- Интеграцию `drawio_convert.py` в цепочку drawio-пути (когда вызывается — до или после MCP).

**Бизнес-правила:**
- Движок выбирается пользователем явно — никакого auto-detect.
- SVG drawio всегда содержит embedded XML (атрибут `content=`) для round-trip редактирования.
- Syntax ссылки (XML vs Markdown) определяется из `.doc-root.yaml`.
- `claude-mermaid` submodule не редактируется — используется только как MCP через `mcpServers`.

**Acceptance Criteria для проверки архитектуры:** AC-001, AC-004, AC-008, AC-009, AC-012.
