# Lessons Learned — gramax marketplace

> Append-only журнал. Каждая запись — короткий урок из реальной работы (фича, bug, surprise). Формат: `## YYYY-MM-DD — <тема>`, далее 3-5 строк: контекст / что узнали / как теперь делаем.

## 2026-05-08 — diagram-on-demand: симметрия mermaid/drawio в Research-фазе

**Контекст:** при декомпозиции фичи с двумя движками первый промпт Researcher'у был перекошен в сторону drawio (исходное предположение, что mermaid закрыт vendored claude-mermaid). Пользователь это заметил.

**Что узнали:** при разведке мультибэкендной фичи RES-промпт должен быть симметричным — равный объём подзадач и критериев сравнения на каждый бэкенд. Иначе вывод смещается.

**Как теперь делаем:** в `pm-agent`-промпте при декомпозиции фичи с N движками/бэкендами явно требовать симметричную RES-таблицу (Инструмент / Тип / Лицензия / Активность / AI-fit / Заметка) на каждый бэкенд + общий compare-блок.

## 2026-05-08 — TDD-stubs могут быть частично зелёными от старта

**Контекст:** QA-author создал 12 stubs; 9 упали (требуют новых артефактов), 3 прошли сразу (тестируют уже существующие `drawio_convert.py`/`slugify.py`).

**Что узнали:** «failing stubs» — не догма. Если AC опирается на pre-existing функциональность, тест может быть зелёным сразу и это валидно: он защищает от регрессии. Главное — каждый stub отображается на номер AC и проверяет семантику, а не просто `[ -f x ]`.

**Как теперь делаем:** в `qa-author-agent`-промпте явно разрешать pre-passing stubs для AC, опирающихся на существующий код; помечать их в README тестов с пометкой «regression guard».
