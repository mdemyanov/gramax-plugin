# Mermaid для Gramax

Skill переводит текстовое описание в корректный Mermaid DSL, записывает его в отдельный `.mermaid`-файл рядом со статьёй и вставляет тег-ссылку в md-документ. Работает без MCP, без скриптов, без preview. Создаёт два артефакта: `.mermaid`-файл (DSL) и тег `<mermaid path="…"/>` в md.

Адаптировано из [axtonliu/axton-obsidian-visual-skills](https://github.com/axtonliu/axton-obsidian-visual-skills) (MIT). Полный текст лицензии и список изменений — в [`LICENSE.upstream.md`](LICENSE.upstream.md).

## When to use

- Пользователь просит визуализировать процесс, архитектуру, цикл, сравнение, иерархию.
- Нужно flowchart / sequence / gantt / class / state / ER / pie / mindmap.
- Вставить mermaid-диаграмму в md-файл Gramax-каталога.

**Не для:** drawio-диаграмм (для drawio используй `gramax:drawio`), preview диаграмм в браузере, рендера в SVG/PNG.

## Fallback при ambiguous-request

Если запрос пользователя не содержит явного engine-keyword («mermaid», «drawio», «.drawio»), спроси пользователя:

> Какой движок использовать для диаграммы?
>
> 1. **mermaid** — файл `.mermaid` рядом со статьёй, тег-ссылка в md; рендер Gramax-фронтендом. Подходит для flowchart, sequence, gantt, ER, state, class, pie, mindmap.
> 2. **drawio** — через внешний плагин `Agents365-ai/drawio-skill`; создаёт `.svg`-файл рядом со страницей. Подходит для сложных схем, BPMN, кастомных стилей.
>
> По умолчанию — **mermaid**, если не укажешь.

После ответа пользователя:
- mermaid → выполняй workflow ниже
- drawio → делегируй в `gramax:drawio`

## Quick start

1. Прочитай запрос пользователя — выдели сущности, отношения, последовательность.
2. Выбери тип диаграммы (см. ниже). По умолчанию — flowchart TB.
3. Определи `target_page` (md-файл статьи). Если не указан — спроси пользователя.
4. Предложи имя `.mermaid`-файла по naming convention (см. раздел «Naming convention»). Запроси подтверждение.
5. Если файл с таким именем уже существует — прочитай его первые 5 строк, сообщи пользователю, предложи три опции: перезаписать / выбрать другое имя / отменить. Без явного «перезаписать» — не трогай файл.
6. Сгенерируй DSL по правилам ниже, выполни самопроверку по checklist.
7. Создай `.mermaid`-файл через **Write tool** в той же директории, что и `target_page`. Файл содержит только чистый DSL (без Gramax-разметки, без обёрток).
8. Вставь тег в md-файл через **Edit tool**:
   ```
   <mermaid path="./<filename>.mermaid" width="<W>px" height="<H>px"/>
   ```
   Тег самозакрывающийся, с пустой строкой до и после.
9. Сообщи пользователю, какие файлы созданы/изменены.

## Naming convention для .mermaid-файлов

Правило: `<page-slug>-<diagram-slug>.mermaid`

### Формирование `<page-slug>`

- Берётся имя md-файла (`target_page`) без расширения.
- **Специальный кейс:** если файл называется `_index.md` — берётся имя родительской директории.
- Примеры:
  - `docs/auth/overview.md` → `page-slug = overview`
  - `docs/auth/_index.md` → `page-slug = auth`
  - `docs/getting-started.md` → `page-slug = getting-started`

### Формирование `<diagram-slug>`

- Извлекается тематическое имя из запроса пользователя: ключевые существительные процесса/системы.
- Kebab-case, только ASCII, не более 30 символов.
- Если slug длиннее 30 символов — обрезать до 30, сохраняя целые слова (не обрезать посреди слова).
- Если пользователь не назвал тему явно — использовать `diagram` как fallback.
- Если тема диаграммы содержит кириллицу — предложить два варианта: семантический перевод на английский и ввод slug вручную пользователем.
- Символы `:`, `*`, `?`, `"`, `<`, `>`, `|`, `\`, пробелы → заменять на `-`.

### Примеры

| md-файл | Тема запроса | Предложенное имя |
|---------|--------------|-----------------|
| `docs/auth/overview.md` | «процесс авторизации» | `overview-auth-flow.mermaid` |
| `docs/auth/_index.md` | «архитектура модуля» | `auth-module-architecture.mermaid` |
| `docs/api/endpoints.md` | (не указана) | `endpoints-diagram.mermaid` |
| `docs/deploy/k8s-setup.md` | «deployment pipeline» | `k8s-setup-deployment-pipeline.mermaid` |
| `docs/main.md` | «очень длинное название процесса обработки данных» | `main-data-processing-pipeline.mermaid` (≤30 символов slug) |

## Gramax-интеграция

Если пользователь указал `target_page` (md-файл) — создай `.mermaid`-файл рядом со статьёй и вставь тег-ссылку.

**Формат тега (одинаков для XML- и Markdown-каталогов):**

```markdown
<mermaid path="./diagram.mermaid" width="800px" height="450px"/>
```

- Тег самозакрывающийся.
- Вставляется с пустой строкой до и после.
- Атрибуты `width` и `height` обязательны.

**Default значения (из ground-truth `blocks.md`):**
- `width="800px"`
- `height="450px"`

Пользователь может переопределить явно: «ширина 1200, высота 600» → `width="1200px" height="600px"`. Если указан только один параметр — второй берётся из default.

**Синтаксис каталога (`.doc-root.yaml`):**

Открой ближайший `.doc-root.yaml` (обход вверх от `target_page`), поле `syntax: XML` или `syntax: Markdown`. Синтаксис каталога влияет на содержимое md-файлов, но **не** на формат тега `<mermaid path="…"/>` — он одинаков для обоих. `.mermaid`-файл содержит только чистый DSL без разметки каталога.

Если `.doc-root.yaml` не найден — продолжи с предупреждением: «`.doc-root.yaml` не обнаружен; тег `<mermaid path="…"/>` вставлен по умолчанию».

## Поддерживаемые типы (Gramax)

| Тип | Подходит для |
|-----|--------------|
| `flowchart TB/LR` | Процессы, workflows, decision tree, AI-агенты |
| `sequenceDiagram` | Взаимодействия, API-вызовы, message flow |
| `gantt` | Планы, расписания, timeline |
| `classDiagram` | UML классы, доменные модели |
| `stateDiagram-v2` | Состояния, lifecycle, переходы |
| `erDiagram` | Сущности БД, отношения, схемы |
| `pie` | Доли, распределение, проценты |
| `mindmap` | Иерархия концепций, разбор темы |

**НЕ поддерживаются в Gramax:** `gitGraph`, `journey`, `requirementDiagram`, `C4Context` — рендер падает. При запросе таких типов сообщи пользователю и предложи замену (например, `C4Context` → `flowchart TB` с подграфами).

## Критические правила синтаксиса

### 1. Конфликт с list-syntax (самая частая ошибка)

Парсер mermaid читает `1. Текст` как Markdown-список и падает с `Unsupported markdown: list`.

```
❌ A[1. Анализ]
✅ A[1.Анализ]                # без пробела
✅ A[① Анализ]                # circled-numbers ① ② ③ ④ ⑤ ⑥ ⑦ ⑧ ⑨ ⑩
✅ A[(1) Анализ]              # скобки
✅ A[Шаг 1: Анализ]           # префикс
✅ A[Анализ]                  # без нумерации
```

### 2. Subgraph с пробелами

```
❌ subgraph Core Process
✅ subgraph core["Core Process"]    # ID + display name
✅ subgraph core_process            # просто ID без пробелов
```

### 3. Ссылки на ноды — только по ID

```
A[Display Text A]
B["Display Text B"]

✅ A --> B                          # по ID
❌ Display Text A --> Display Text B
```

### 4. Спецсимволы в тексте ноды

- Пробелы → оборачивай в кавычки: `A["Текст с пробелами"]`
- Кавычки внутри → заменяй на `『』`
- Скобки внутри → заменяй на `「」`
- Переносы строки `<br/>` работают только в круглых нодах: `((Текст<br/>Перенос))`
- Кириллица — оборачивай в кавычки: `A["Старт"]`

### 5. Стрелки

- `-->` сплошная
- `-.->` пунктирная (опциональные пути, обратная связь)
- `==>` жирная (акцент)
- `~~~` невидимая (только для лейаута)
- `<-->` двунаправленная

Полный справочник синтаксиса, troubleshooting и edge cases — в [`references/syntax-rules.md`](references/syntax-rules.md).

## Конфигурация (опции для пользователя)

**Направление:**
- `TB`/`TD` (top→bottom, default), `BT`, `LR` (left→right, для timeline), `RL`

**Уровень детализации:**
- `simple` — только ключевые ноды
- `standard` — баланс деталей и читаемости (default)
- `detailed` — полные аннотации
- `presentation` — для слайдов: крупный текст, меньше нод

**Стиль:**
- `minimal` — монохром
- `professional` — семантические цвета, иерархия (default)
- `colorful` — яркие цвета, высокий контраст
- `academic` — формальный для документации

## Типовые паттерны

### Swimlanes (группировка)

```
flowchart TB
    subgraph core["Основной процесс"]
        A --> B --> C
    end
    subgraph support["Вспомогательные системы"]
        D
        E
    end
    core -.-> support
```

### Цикл с обратной связью

```
flowchart TB
    A[Старт] --> B[Обработка]
    B --> C[Результат]
    C -.->|Обратная связь| A
```

### Hub-and-spoke

```
flowchart TB
    Hub[Центр]
    A[Узел 1] --> Hub
    B[Узел 2] --> Hub
    C[Узел 3] --> Hub
```

## Workflow генерации

1. **Понять контент** — сущности, иерархия, последовательность, контрасты.
2. **Выбрать тип** — flowchart по умолчанию, если неоднозначно.
3. **Применить конфиг** — параметры из запроса + sensible defaults.
4. **Сгенерировать DSL** — соблюдая 5 критических правил.
5. **Самопроверка** (см. checklist ниже).
6. **Определить имя файла** — по naming convention, запросить подтверждение.
7. **Проверить коллизию** — если файл существует, прочитать первые 5 строк, показать пользователю, запросить: перезаписать / другое имя / отменить.
8. **Определить Gramax-syntax** — открой `.doc-root.yaml` (для контекста каталога).
9. **Создать `.mermaid`-файл** через Write tool — только чистый DSL, без обёрток.
10. **Вставить тег** через Edit tool — самозакрывающийся тег с пустыми строками до/после.
11. **Сообщить результат** — перечислить созданные/изменённые файлы.

## Checklist перед записью в файл

- [ ] Нет паттерна `число. пробел` в тексте нод
- [ ] Subgraph с пробелами в имени → ID+display
- [ ] Все ссылки на ноды — по ID, не по display-тексту
- [ ] Стрелки — валидный синтаксис (`-->`, `-.->`, `==>`)
- [ ] Тип входит в 8 поддерживаемых Gramax (или предупреждение пользователю)
- [ ] Кириллица в node-label обёрнута в кавычки: `A["Старт"]`
- [ ] Пустые строки до и после тега в md
- [ ] **Без emoji** в тексте нод — используй цвет или текст вместо
- [ ] Стили задекларированы консистентно
- [ ] Содержимое `.mermaid`-файла не содержит Gramax-разметки: нет `<mermaid>`, нет ` ```mermaid ` — только чистый DSL

## Backward compatibility

При чтении `target_page` проверяй наличие inline-блоков устаревшего формата:

**Обнаружен inline-блок `<mermaid>…</mermaid>`:**

```
Обнаружен inline-блок <mermaid>…</mermaid> в файле — это устаревший формат.
Gramax ожидает тег-ссылку <mermaid path="…"/>.
Предлагаю миграцию: извлечь DSL в файл <page-slug>-existing.mermaid и заменить блок на тег.
Выполнить миграцию? (да / нет / пропустить и создать новую диаграмму)
```

**Обнаружен fenced block ` ```mermaid … ``` `:**

Аналогично — предупреждение с подстрокой «устаревший формат» и предложением миграции.

**Без подтверждения пользователя — файл не трогать.** Skill не выполняет миграцию молча.

**Обнаружен файл `.mermaid` с предложенным именем:**

1. Прочитай первые 5 строк существующего файла.
2. Выведи содержимое пользователю.
3. Предложи три опции: перезаписать / выбрать другое имя / отменить.
4. Без явного «перезаписать» — не трогай файл.
